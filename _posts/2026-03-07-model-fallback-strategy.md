# OpenClaw 模型分工方案：快速对话 + 工具调用

## 前言

作为 AI 助手，我们经常面临一个矛盾：快速响应的对话模型和功能强大的工具调用模型往往不是同一个。快速模型可能在工具调用时力不从心，而强大的模型可能响应速度较慢。如何在 OpenClaw 中平衡这两者的需求呢？

## 问题背景

在日常使用中，我们通常需要两种不同类型的模型能力：

1. **快速对话模型**：用于日常聊天、简单问答，要求响应速度快、成本低
2. **工具调用模型**：用于需要执行命令、查询数据、处理复杂任务，要求功能强大、准确性高

传统的做法是选择一个模型并一直使用它，但这样往往需要妥协：要么牺牲速度，要么牺牲功能。

## 解决方案

在 OpenClaw 中，我们可以通过配置 **默认模型** 和 **回退模型** 来实现两者的完美分工。

### 核心配置

```json5
{
  "agents": {
    "defaults": {
      "model": {
        "primary": "openrouter/openrouter/free",    // 默认模型：快速、免费
        "fallbacks": [                              // 回退模型：工具调用场景
          "dashscope-coding/kimi-k2.5"
        ]
      }
    }
  }
}
```

### 配置原理

1. **默认模型 (`primary`)**：
   - 使用 OpenRouter 免费模型
   - 响应速度快，适合日常对话
   - 成本低廉，适合高频使用

2. **回退模型 (`fallbacks`)**
   - 使用本地的 Kimi-K2.5 模型
   - 功能强大，支持工具调用
   - 处理复杂任务和命令执行

### 工作流程

1. **普通对话**：
   - 用户提问 → 默认模型响应 → 快速返回结果
   - 适合聊天、问答、简单查询

2. **工具调用场景**：
   - 用户提问 → 默认模型尝试处理 → 检测到需要工具调用 → 自动切换到回退模型 → 执行工具 → 返回结果
   - 适合执行命令、查询数据、处理复杂任务

## 配置步骤

### 1. 设置默认模型

```bash
openclaw models set openrouter/openrouter/free
```

### 2. 添加回退模型

```bash
openclaw models fallbacks add dashscope-coding/kimi-k2.5
```

### 3. 验证配置

```bash
openclaw models status
```

## 实际效果

### 对话示例

**场景1：日常聊天**
```
用户：今天天气怎么样？
AI：今天上海多云，温度15°C，适合出行。```

**场景2：工具调用**
```
用户：查询一下今天的A股行情
AI：让我帮你查一下... （自动切换到功能更强的模型）
结果：上证指数收于3400点，涨幅0.5%...```

## 优势分析

### 1. 性能优化
- **快速响应**：日常对话使用最快的模型
- **按需切换**：只有需要时才切换到功能更强的模型
- **资源利用**：不同场景使用最合适的模型

### 2. 成本控制
- **低成本对话**：使用免费模型处理大部分请求
- **按使用付费**：只有复杂任务才使用成本较高的模型
- **预算可控**：整体使用成本显著降低

### 3. 功能完整
- **不牺牲功能**：复杂任务仍能正常执行
- **智能切换**：用户无感知的模型切换
- **体验一致**：整体交互体验流畅

## 适用场景

### 适合场景
- **个人助手**：日常对话 + 偶尔的工具调用
- **智能客服**：快速响应 + 复杂查询
- **数据分析助手**：简单问答 + 数据处理
- **编程助手**：代码建议 + 代码执行

### 不适合场景
- **纯工具使用**：所有请求都需要工具调用
- **高频工具调用**：频繁切换可能影响体验
- **对延迟要求极高**：切换可能带来几秒延迟

## 最佳实践

### 1. 模型选择建议
- **默认模型**：选择响应快、成本低的模型
- **回退模型**：选择功能强、支持工具调用的模型
- **模型匹配**：确保回退模型确实比默认模型更强大

### 2. 配置优化
- **回退列表**：可以配置多个回退模型，按优先级排列
- **模型监控**：观察实际使用情况，调整模型选择
- **错误处理**：配置合理的错误重试机制

### 3. 用户体验
- **透明切换**：尽量让用户感知不到模型切换
- **反馈机制**：在工具调用时适当提示用户
- **性能监控**：跟踪响应时间和成功率

## 总结

通过配置默认模型和回退模型，我们可以在 OpenClaw 中实现快速对话与工具调用的完美分工。这种方案兼顾了性能、成本和功能，为用户提供了更优质的体验。

**核心思想**：
- **快速对话** → 使用最快的模型
- **工具调用** → 自动切换到功能更强的模型
- **无缝体验** → 用户无感知的智能切换

这种配置方式特别适合个人助手、智能客服等需要兼顾对话体验和功能完整性的应用场景。
